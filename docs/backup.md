# Backup & Restore

Database and media backup/restore for the stack. A backup is a **timestamped directory**
under `dumps/` (gitignored) created by `scripts/dump_all.sh` and consumed by
`scripts/restore_all.sh`. Both are driven through `make`.

The important design choice: **media is in S3/MinIO, not on the container filesystem**, so
it is dumped and loaded *through the app* (Django management commands streaming a tar to/from
the bucket), never by tarring a local directory.

## Quick start

```bash
make dump                  # database + manifest only
make dump media            # database + manifest + all S3/MinIO media (slower)
make dev dump media        # same against the dev stack

make restore dumps/<ts>    # restore database + media + rewrite embedded URLs
make dev restore dumps/<ts>
```

`make dev …` runs the management commands inside the dev container (with
`--workdir /app/backend`); plain `make …` targets the production stack. Both read Postgres
credentials from `.env`.

## What a backup contains

A backup directory `dumps/<YYYY-MM-DD_HH-MM-SS>/` holds:

| File | Contents | Always? |
|------|----------|---------|
| `database.sql.gz` | `pg_dump --clean --if-exists --no-owner --no-privileges`, gzipped | yes |
| `manifest.json`   | `media_base_url`, `object_count`, `total_bytes` of the bucket | yes |
| `media.tar.gz`    | every S3/MinIO object **except** the `static/` prefix | only with `media` |

`static/` is excluded because it is regenerated by `collectstatic` and needs no backup.

## Management commands

These live in the `common` app (`backend/apps/common/management/commands/`) and operate on
`default_storage` (the S3/MinIO bucket):

| Command | Purpose |
|---|---|
| `media_info` | print the JSON manifest (media base URL + counts) to stdout |
| `media_dump` | stream a gzipped tar of all media objects to stdout |
| `media_load` | read a gzipped tar from stdin and upload each object to the bucket |
| `rewrite_media_urls --from <url>` | replace an old media base URL with the current one across all text fields |

They are usable standalone, e.g.:

```bash
docker compose exec -T backend python manage.py media_dump --prefix users/ > users-media.tar.gz
cat users-media.tar.gz | docker compose exec -T backend python manage.py media_load --skip-existing
```

## Restoring into a *different* S3/MinIO

This is the reason for the manifest. `FileField`/`ImageField` values store only the storage
**key**, so their URLs are rebuilt automatically for whatever S3/MinIO is configured — they
need no rewriting. But media embedded **inside HTML/text content** (rich-text bodies,
descriptions, …) holds *absolute* URLs like `https://old-host/old-bucket/...`.

So after loading media into a new bucket, restore reads the backup's `media_base_url` from
`manifest.json` and runs `rewrite_media_urls --from <old_base>`, replacing it with the
current storage base across every `CharField`/`TextField`. A restore into a new
bucket/domain keeps every in-content image and file working.

Run the rewrite on its own (with `--dry-run` to preview):

```bash
docker compose exec -T backend python manage.py rewrite_media_urls \
  --from https://old/bucket/ --to https://new/bucket/ --dry-run
```

## Automated database backups (optional)

The `make dump` flow above is **manual and on-demand**. For scheduled, retained,
optionally offsite DB backups in production, `docker-compose.prod.yml` ships an optional
`db-backup` service ([tiredofit/db-backup](https://github.com/tiredofit/docker-db-backup)),
**disabled by default** behind the `backup` compose profile.

**Scope: database only.** Media stays on the tooling above plus S3/MinIO bucket
versioning/replication — this service never touches it.

Enable it by activating the profile in `./.env`:

```bash
COMPOSE_PROFILES=backup      # then: make prod deploy
# or one-off: docker compose -f docker-compose.prod.yml --profile backup up -d
```

Configure via `.env` (full list in `.env.example`). It reuses the app's `POSTGRES_*`
credentials and connects to the `db` service:

| Variable | Purpose | Default |
|---|---|---|
| `DB_BACKUP_INTERVAL` | minutes between backups | `1440` (24h) |
| `DB_BACKUP_CLEANUP_TIME` | delete backups older than N minutes | `10080` (7d) |
| `DB_BACKUP_LOCATION` | `FILESYSTEM` or `S3` | `FILESYSTEM` |
| `DB_BACKUP_FS_PATH` | host path for filesystem dumps | `./db-backups` |
| `DB_BACKUP_S3_BUCKET` / `_PATH` / `_HOST` / `_REGION` / `_KEY_ID` / `_KEY_SECRET` | S3 target (own bucket/prefix, separate from media) | — |

**Restore** a produced dump (a compressed `pg_dump`) into the running database:

```bash
zstd -dc db-backups/<file>.sql.zst | \
  docker compose -f docker-compose.prod.yml exec -T db psql -U "$POSTGRES_USER" -d "$POSTGRES_DB"
```

**When to skip it:** on **managed Postgres** (RDS / Cloud SQL / …) use the provider's
automated backups + PITR instead — this sidecar targets a self-hosted `db` container.

## Notes

- `dumps/` is gitignored; move backups off the host (object storage, another server) for
  real durability.
- The database restore is **in place** via the dump's `--clean --if-exists` statements
  (`psql -v ON_ERROR_STOP=1`) — it overwrites the current schema/data. There is no full
  `DROP DATABASE`. There is **no confirmation prompt**, so double-check the target stack.
- The media tar streams object-by-object through the app. For very large production buckets,
  mirroring bucket-to-bucket with the MinIO client (`mc mirror`) is faster; the
  `rewrite_media_urls` step still applies afterwards.
- A DB-only `make dump` still calls `media_info`, which lists the bucket — so storage must be
  reachable. See [docs/uv.md](uv.md) for the env workflow and `.env.example` for the
  `DJANGO_MINIO_*` variables.
